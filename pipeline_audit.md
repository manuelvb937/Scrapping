# Pipeline Audit

## 1. Executive Summary
This pipeline is a comprehensive social listening tool designed to scrape Japanese X (Twitter) posts from Yahoo Realtime, clean the text, and group similar posts into subtopics. It dynamically strips out the original search terms so that clustering focuses on the actual discussion topics rather than the shared keywords. It then uses a local transformer model to create semantic embeddings, clusters them using UMAP and HDBSCAN, and finally sends the data to an LLM (OpenAI or Gemini) to assign sentiment to individual posts and generate human-readable topic labels for each cluster.

## 2. Execution Flow
The pipeline is orchestrated via `cli.py` and runs in four main phases:

1. **Scraping (`cli.py scrape`)**: 
   - Calls `src/collector/yahoo_realtime.py:scrape_yahoo_realtime()`.
   - Uses a headless Selenium Chrome browser to scroll and scrape Yahoo Realtime.
   - Deduplicates records and saves them to `data/raw/` in `.jsonl` format.
2. **Preprocessing (`cli.py preprocess`)**: 
   - Calls `src/preprocessing/pipeline.py:preprocess_raw_file()`.
   - Deduplicates again, cleans basic text (`clean_text`), detects language, and extracts hashtags.
   - Saves to `data/processed/`.
3. **Analysis (`cli.py analyze`)**: 
   - Calls `src/analysis/pipeline.py:run_analysis()`.
   - `src/analysis/text_preparation.py:prepare_clustering_texts()`: Strips search queries, URLs, and timestamps to create a clean `clustering_text`.
   - `src/analysis/embeddings.py:TextEmbedder.embed_texts()`: Converts text into dense vectors.
   - `src/analysis/clustering.py:cluster_embeddings()`: Reduces dimensions and groups vectors into clusters.
   - `src/analysis/llm_labeling.py:LLMTopicSentimentLabeler.analyze_posts_batch()`: Asks an LLM for the sentiment of each individual post.
   - `src/analysis/llm_labeling.py:LLMTopicSentimentLabeler.analyze_cluster()`: Asks an LLM to label the overall topic and sentiment of each cluster.
   - Saves `clusters.json` and `analysis.json` to `data/reports/`.
4. **Reporting (`cli.py report` / `export`)**:
   - Calls `src/reporting/generator.py` and `exporter.py`.
   - Generates Markdown, HTML, and CSV reports for manual inspection.

## 4. Clustering Explanation
- **What data is clustered?** The embeddings of `clustering_text` are clustered. `clustering_text` is explicitly generated by stripping the user's search queries, usernames, and URLs (`text_preparation.py:prepare_clustering_text()`).
- **Are raw texts clustered?** No, embeddings are created first.
- **What embedding is used?** Dense semantic vectors created by the Hugging Face model `BAAI/bge-m3` via the `sentence-transformers` library.
- **How is the number of clusters decided?** It is decided automatically by HDBSCAN. HDBSCAN finds areas of high density; it does not require a pre-defined number of clusters (like KMeans does).
- **How are representative posts selected?** In `text_preparation.py:select_representative_posts()`, the code scores posts to prefer medium-length texts. It then iterates through them and uses a Jaccard-like overlap check on character trigrams to ensure it doesn't pick duplicate-sounding posts.
- **How are cluster labels generated?** They come from an LLM. The system sends the representative posts, the top keyword frequencies, and the ignored search terms to OpenAI/Gemini, explicitly prompting it: *"Do NOT use the overall search term as the topic label — instead, identify what makes THIS group of posts different."*

## 5. Sentiment Analysis Explanation
- **Is sentiment assigned per post or per cluster?** Both. `analyze_posts_batch()` assigns sentiment per post, and `analyze_cluster()` assigns an overall sentiment to the topic.
- **What labels are used?** Strictly `positive`, `negative`, or `neutral`.
- **Is confidence calculated?** No. The LLM simply returns the string label via JSON schema. There is no probability/confidence score attached.
- **Does it distinguish sentiment, emotion, and stance?** No, it only evaluates a flat positive/negative sentiment scale.
- **Does it handle Japanese SNS/fandom slang?** Yes. The LLM prompts specifically ask for Japanese cultural awareness (sarcasm, understatement). Furthermore, if the LLM API fails, a hardcoded heuristic fallback in `llm_labeling.py:_heuristic_sentiment()` checks for Japanese fandom slang like `"神"` (God/Awesome), `"推し"` (Favorite/Bias), `"炎上"` (Flaming/Scandal), and `"クソ"` (Shit).

## 6. Use of Transformers / LLMs
- **Currently this pipeline uses Transformers:** Yes, it uses the `sentence-transformers` library to run `BAAI/bge-m3` locally for text embeddings. 
- **Currently this pipeline uses LLMs:** Yes, it uses API calls (via standard HTTP requests) to OpenAI (`gpt-4.1-mini`) or Gemini (`gemini-2.5-flash`) for inference.
- **Currently this pipeline uses other algorithms:** It uses UMAP and HDBSCAN directly rather than relying on a wrapper library like `BERTopic`.

## 7. Fine-tuning Status
- **Are models fine-tuned?** The `BAAI/bge-m3` model is a pre-trained model downloaded from Hugging Face. It is used "as-is" (zero-shot). The LLMs (OpenAI/Gemini) are accessed via standard API prompting. 
- **Does this repo perform fine-tuning?** No. This repository contains absolutely no code for training or fine-tuning models. It is strictly an inference and data processing pipeline.

## 8. Problems / Limitations
1. **Expensive and Slow Sentiment Analysis:** Using a cloud LLM to classify *every single post* via batching is slow (the code includes `time.sleep(10)` to avoid rate limits) and will cost money at scale.
2. **Greedy Clustering Fallback:** If a dataset has fewer than 20 posts, `clustering.py` bypasses HDBSCAN entirely and uses a legacy greedy cosine-similarity threshold, which lacks noise detection.
3. **No Temperature Control:** The LLM API payloads do not explicitly set `temperature=0`, meaning the LLM could theoretically return slightly different topic labels for the exact same cluster on different runs.
4. **No Confidence Thresholds:** Because sentiment is decided by an LLM JSON output rather than a classifier's softmax layer, you cannot filter out "low confidence" sentiment predictions.

## 9. Recommended Next Changes

1. **Add a Transformer-based Sentiment Classifier (High Priority):** 
   - **How:** Replace the `_batch_sentiment_llm` calls with a local Hugging Face model. You can use the `transformers` library pipeline: `pipeline("text-classification", model="kohei-y/kobart-sentiment-jp")`.
   - **Why:** This runs locally, is free, operates instantly, and gives you a confidence score (e.g., 0.89 Positive).
2. **Use LLMs only for Cluster Labeling or Ambiguous Cases (Medium Priority):**
   - **How:** If the local sentiment classifier returns a confidence below `0.60`, *then* route that specific post to the LLM for a second opinion. Keep the LLM for `analyze_cluster()` since generative topic labeling is exactly what LLMs excel at.
3. **Make the Output More Reproducible (Quick Win):**
   - **How:** Add `"temperature": 0.0` to the JSON payloads in `_analyze_with_openai` and `_analyze_with_gemini`. Pass a static `random_state` or seed to HDBSCAN to ensure the clustering results are completely deterministic.
4. **Sentence-transformer Multilingual Embeddings:**
   - Note: You have *already* successfully implemented this by using `BAAI/bge-m3`. This is currently working exactly as intended!

## 10. Beginner-Friendly Explanation
Imagine you want to know what people on Twitter are saying about a new movie. If you just search the movie's name, you get thousands of tweets. Reading them all is impossible.

This pipeline automates reading them. First, a robot (the scraper) goes to Yahoo and copies all the tweets into a file. Then, it "cleans" them—it deletes the movie's name from the text. Why? Because *every* tweet has the movie's name, and if we leave it in, a computer will just say "Wow, these tweets are all about the movie!" We want to know *what else* they are talking about (e.g., the soundtrack, the ending, the actors).

Next, it uses a **Transformer** (a smart math model) to turn each tweet into a list of numbers (an embedding) that represents its meaning. If two tweets have similar numbers, they mean similar things. We plot these numbers on a graph and use **Clustering** (UMAP and HDBSCAN) to draw circles around groups of tweets that are clumped together. 

Finally, the pipeline hands a few tweets from each circle to an **LLM** (like ChatGPT). It asks the AI, "Read these five tweets. What is this group talking about, and are they happy or angry?" The AI replies with a neat label like "Praising the Soundtrack: Positive," and the pipeline saves a beautiful report for you to read.
